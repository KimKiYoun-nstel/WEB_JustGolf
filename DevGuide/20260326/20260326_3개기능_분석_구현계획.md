# 20260326 기능 보완(3건) 분석 및 구현 계획

## 1) 요구사항 요약

### 1-1. 종료 대회 UX/결과보기 추가
- 사용자 대회 목록에서 `종료(done)` 대회는 더 이상 `참가 신청/참가정보 수정` 진입을 활성화하지 않는다.
- 대회 목록에서 `종료(done)` 대회의 `라운드 관리`도 비활성화(또는 결과보기로 대체)한다.
- `결과보기` 페이지를 신규 추가한다.
- 결과는 `DevGuide/20260326/단체 스코어.pdf` 기반으로 구성한다.
- 결과 데이터는 사용자 표시 이름(이름) 기준이며, 서비스 키는 닉네임이므로 아래 매칭 규칙을 적용한다.
  - `profiles.full_name`이 있고, 동일 이름 사용자가 1명인 경우 동일인으로 간주해 본인 표시
  - 동명이인/이름 미입력 사용자는 자동 본인 표시 보류
- 결과 페이지에서 `대회 결과 + 기존 조편성`을 함께 확인 가능해야 한다.
- 종료 대회(`done`)는 결과 포함 관련 운영 정보가 수정되지 않도록 잠금 처리한다.
- 결과보기 진입 시 `DevGuide/20260326/summary.txt` 내용을 팝업으로 노출하고 `오늘 하루 보지 않기`를 제공한다.
  - 단, 사용자는 최소 1회 이상은 반드시 팝업을 보게 한다.

### 1-2. 라이브 조편성 추첨 개선
- 신규 대회 라이브 조편성 시작 시, `종료된 특정 대회`를 참조 대회로 선택할 수 있어야 한다.
- 참조 대회의 조편성 데이터를 기반으로, 같은 조였던 인원이 다음 대회에서도 연속으로 같은 조가 되지 않도록 추첨 로직을 개선한다.

### 1-3. user-XXX 사용자 정리 + 온보딩 재강제
- 닉네임이 `user-XXX` 패턴인 사용자를 삭제한다.
- 로그인 시 온보딩을 다시 1회 강제한다.
- 온보딩 문구 추가: `카카오톡방 닉네임과 동일한 닉네임으로 수정해주세요.`
- 온보딩 필수 입력: `닉네임, 이름, 전화번호`
- 기존 보유 정보는 입력칸에 사전 채움
- 저장 버튼 클릭 시 확인 팝업을 추가해 반드시 확인 후 저장

### 사용자/관리자/실패 시나리오 요약
- 사용자 목표: 종료 대회는 신청 불가를 명확히 인지하고, 결과/조편성을 읽기 전용으로 확인
- 관리자 목표: 종료 대회의 운영 데이터는 잠금, 신규 대회 추첨은 과거 조편성 중복을 자동 완화
- 실패 시나리오: PDF 자동 파싱 실패, 이름 동명이인 오탐, user-XXX 삭제 시 연관 데이터 손실 위험

## 3) 데이터/권한(RLS) 영향

### 3-1. 결과 데이터 모델
- 권장: 결과를 구조화 저장하는 테이블 신설
  - 예: `tournament_result_rows`, `tournament_result_notices`
- 핵심 컬럼
  - `tournament_id`, `row_order`, `display_name`, `score_payload(jsonb)`, `source`
  - `matched_user_id`, `matched_registration_id`, `match_status(matched/ambiguous/pending)`
- `summary.txt`는 공지 테이블로 보관하거나, 페이지 전용 정적 리소스+버전키로 관리

### 3-2. 종료 대회 수정 잠금
- 대상 쓰기 영역
  - `registrations`, `side_events`, `side_event_registrations`, `tournament_groups`, `tournament_group_members`, `draw_sessions`, `draw_events`
- 권장 방안
  - 앱 UI 비활성화 + 서버/API 가드 + DB(트리거 또는 정책) 3중 잠금
  - 특히 `draw` API는 Service Role 사용 중이므로 서버단 상태 검증 필수

### 3-3. 추첨 참조 대회 권한
- 참조 대회 조회는 인증 사용자 SELECT 범위 내
- 추첨 실행/변경은 기존 관리자 권한 유지
- 참조 대회가 `done`이고 조편성 데이터가 있는 경우만 선택 가능

### 3-4. user-XXX 삭제 영향
- 삭제는 `auth.users` 기준으로 수행(연쇄 삭제 발생 가능)
- 리스크
  - `registering_user_id` FK(on delete cascade)로 신청 데이터 삭제 가능성
- 실행 전 필수
  - 삭제 대상 건수/연관 데이터 영향 Dry-run 쿼리
  - 백업/스냅샷 확보

### 3-5. 온보딩 필수화
- `middleware.ts`의 온보딩 완료 판단식 강화
  - 닉네임(user- 금지) + 이름 + 전화번호 필수
- 온보딩 저장 시
  - `profiles.full_name` + `auth.users.user_metadata.phone` 동시 정합성 유지
  - 기존 값 사전 채움

### 관리자/일반 사용자/비로그인 플로우 체크
- 관리자
  - 종료 대회는 결과 조회만 가능, 수정 액션 차단
  - 신규 대회 추첨 시 참조 대회 선택 가능
- 일반 사용자
  - 종료 대회 참가 신청 불가(버튼/경로/DB 모두)
  - 결과보기 읽기 전용 접근 가능
- 비로그인
  - 기존 미들웨어 정책 유지(로그인 필요)

## 5) 단계별 구현 계획

### Phase 0. 결정 필요 사항 정리
- PDF가 이미지 기반으로 텍스트 추출 도구 부재(현 환경 `pdftotext/gs` 없음)
- 선택지
  1. 수동 정제 CSV/JSON 작성 후 결과 테이블 적재
  2. 별도 OCR 파이프라인 도입 후 자동 적재
- 이번 범위 권장: 1번(일정/안정성 우선), OCR은 후속 과제

### Phase 1. DB/RLS/데이터 준비
- 결과 테이블 + 공지(요약문) 저장 구조 추가
- draw 세션에 참조 대회 정보 저장 컬럼 추가
- 종료 대회 write lock(트리거/정책) 추가
- user-XXX 정리 SQL 및 온보딩 재강제 SQL 작성
- 모든 SQL은 `db/migrations/*.sql`로 작성 후 SQL Editor 수동 실행

### Phase 2. API/권한
- `draw` API
  - `start_session`에 `referenceTournamentId` 수용
  - 대상/참조 대회 유효성 검증 + done 잠금 검증
  - 조 배정 시 같은 조 이력 페널티 반영
- 결과 조회 API(또는 클라이언트 조회 경로) 구성
  - 결과/조편성/본인표시 플래그 전달

### Phase 3. UI
- 대회 목록
  - done 대회: 참가 신청/라운드관리 비활성화, 결과보기 버튼 제공
- 결과보기 페이지
  - 결과 테이블 + 기존 조편성 병행 노출
  - 본인 표시 배지(유니크 full_name 매칭)
  - `summary.txt` 팝업 + `오늘 하루 보지 않기` + 최소 1회 강제
- 라이브 추첨 화면
  - 참조 done 대회 선택 UI
  - 적용 상태/매칭률/주의 안내 표시
- 온보딩 화면
  - 안내 문구 추가
  - 닉네임/이름/전화번호 필수
  - 저장 전 확인 팝업

### Phase 4. 검증/문서
- `npm run build` 필수 통과
- 영향 범위 unit/e2e 실행
- DevGuide 체크리스트 업데이트(요구사항 누락 0)

## 6) 수동 QA 시나리오 (3~5)
1. 종료(done) 대회 카드에서 `참가 신청`, `라운드 관리`가 비활성/미노출되고 `결과보기`로만 진입되는지 확인
2. 결과보기 최초 진입 시 요약 팝업이 반드시 뜨고, `오늘 하루 보지 않기` 후 같은 날 재진입 시 미노출되는지 확인
3. 결과보기에서 본인 표시가 `full_name 유니크 매칭` 사용자만 표시되고 동명이인/이름 미입력자는 보류되는지 확인
4. 라이브 추첨 시작 시 참조 대회 선택 후 같은 조 이력이 많은 조합이 후순위로 밀리는지(중복 완화) 확인
5. 기존 사용자 로그인 시 온보딩 재진입, 필수값 누락 저장 차단, 확인 팝업 승인 후 저장 성공 확인
