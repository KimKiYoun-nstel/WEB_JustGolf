# 20260326 결과데이터 DB구조 개선 분석 및 구현 계획

## 1) 요구사항 요약
- 결과 보기 페이지는 PDF를 보기 좋게 보여주는 수준을 넘어서, 선수별 상세 스코어와 경기 내역을 신뢰할 수 있게 조회해야 한다.
- 현재처럼 코드 내부 fallback 데이터에 의존하면 오탈자와 검증 누락을 발견하기 어렵다.
- `tournament_result_rows`는 자유형 결과 표시에는 유용하지만, 선수별 상세 스코어와 18홀 데이터 관리에는 구조가 느슨하다.
- 장기적으로는 대회별 결과 데이터를 DB에 구조화해 저장하고, 결과 페이지는 DB를 기준으로 검색/정렬/요약을 수행해야 한다.
- 결과 데이터는 운영 특성상 “초기 입력/검수 이후에는 사실상 수정되지 않는 읽기 전용 스냅샷”으로 취급하는 것이 맞다.

## 2) 현재 구조 및 확인 결과
- 현재 결과 페이지 데이터는 완전한 DB 기반이 아니다.
- 조회 우선순위
  - `tournament_result_assets`: summary, pdf_url
  - `tournament_result_rows`: 이름, 합계, 비고, 얕은 payload
  - `lib/results/legacyTournamentResultDetails.ts`: 홀별 상세, 경기 내역 fallback
- API 기준 코드는 [results API](D:/CodeDev/WEB_JustGolf/app/api/tournaments/[id]/results/route.ts) 이다.
- 현재 PDF는 텍스트 추출형이 아니라 이미지형이다.
  - `pypdf`, `fitz` 로 텍스트 추출 시 페이지 텍스트가 비어 있었다.
  - 현재 로컬 환경에는 OCR 도구가 없어서 자동 신뢰 검증이 불가능하다.
- 따라서 현재 fallback 상세 데이터는 “PDF 자동 추출 결과”가 아니라 “수동 정리본”으로 보는 것이 맞다.
- 사용자 제보 기준 `주창연 -> 주창언` 불일치가 실제로 존재했고, 이번 수정에서 fallback과 적재 SQL을 함께 보정했다.

## 4) 데이터/권한(RLS) 영향
- 기존 정책 유지
  - 인증 사용자는 결과 조회 가능
  - 관리자만 초기 입력 가능
- 신규 구조도 동일 원칙 유지
  - `tournament_result_import_batches`: 관리자 초기 입력/검수, 인증 사용자 read
  - `tournament_result_scorecards`: 관리자 초기 입력/검수, 인증 사용자 read
  - `tournament_result_scorecard_holes`: 관리자 초기 입력/검수, 인증 사용자 read
- Service Role은 계속 서버 API 내부에서만 사용하고, 클라이언트에는 노출하지 않는다.
- 공개 이후에는 결과 데이터를 일반 CRUD 대상으로 보지 않고, 정정이 필요할 때만 관리자 검수 절차로 제한하는 방향이 적절하다.

## 5) 목표 구조
### 5-1. 유지할 테이블
- `tournament_result_assets`
  - 대회별 summary, pdf_url, 갈무리 텍스트 관리
- `tournament_result_rows`
  - 자유형 섹션, 간단 요약, 레거시 호환 용도 유지

### 5-2. 추가할 테이블
- `tournament_result_import_batches`
  - 어떤 파일/방식으로 결과를 적재했는지 이력 관리
  - `source_kind`, `source_file_name`, `source_checksum`, `source_note`, `payload`
- `tournament_result_scorecards`
  - 선수별 결과 1건 = 1행
  - 순위, 합계, 코스, 핸디, 니어, 롱기, 검증상태, 매칭상태 저장
  - `registration_id`, `matched_user_id`, `source_row_id` 로 기존 데이터와 연결
- `tournament_result_scorecard_holes`
  - 18홀 상세 저장
  - `scorecard_id`, `nine_type`, `hole_no`, `score`, `par`, `result_label`, `payload`

### 5-3. 왜 이렇게 나누는가
- 결과 페이지의 검색/정렬/상세/요약은 선수 단위 데이터가 핵심이다.
- 홀별 상세는 반복 구조이므로 별도 테이블로 분리해야 조회와 수정이 명확하다.
- PDF 형식은 대회마다 달라도, 최종 저장 구조는 동일하게 유지할 수 있다.
- 수동 입력, OCR 결과, 외부 업로드 모두 `import_batch -> scorecard -> holes` 흐름으로 흡수할 수 있다.
- 결과 데이터가 읽기 위주라서, 입력 자유도보다 조회 일관성과 검증 이력 추적이 더 중요하다.
- 즉, “자주 수정되는 업무 테이블”이 아니라 “검수 후 고정되는 결과 스냅샷 테이블”로 설계해야 한다.

## 6) 조회 전략 제안
### 6-1. 최종 목표
- 결과 페이지 조회 우선순위
  1. `tournament_result_scorecards` + `tournament_result_scorecard_holes`
  2. `tournament_result_rows`
  3. fallback 없음

### 6-2. 전환 원칙
- fallback 코드는 점진적으로 제거한다.
- 특정 대회에 structured scorecard 데이터가 1건 이상 존재하면, 결과 페이지는 structured 데이터만 사용한다.
- `tournament_result_rows`는 결과 상세의 원천이 아니라, 자유형 섹션 또는 레거시 데이터로만 취급한다.
- 결과 페이지는 조회 최적화가 목표이므로, 조인/가공을 최소화한 read model 형태로 유지하는 것이 적절하다.
- 결과 확정 이후에는 update/delete를 상시 허용하기보다, 검수 단계에서만 제한적으로 허용하는 운영 모델이 맞다.

## 7) PDF 추출 신뢰성 대응
- 현재처럼 이미지형 PDF는 “자동 추출 후 바로 노출” 방식이 위험하다.
- 권장 흐름
  1. 원본 PDF 업로드 또는 링크 등록
  2. 관리자 수동 입력 또는 OCR 임시 적재
  3. 관리자 검수 후 `verification_status = verified`
  4. 결과 페이지는 verified 데이터 우선 노출
- 즉, OCR이 들어가더라도 최종 공개 전에 검수 단계가 반드시 필요하다.
